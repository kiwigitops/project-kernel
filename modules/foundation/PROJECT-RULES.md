# Project Rules

These are the non-negotiable operating rules for this project. Keep them short enough that contributors actually read them.

## Working Rules

- Prefer clear, boring solutions before clever ones.
- Keep changes scoped to the stated task unless a wider change is required for correctness.
- Update documentation when behavior, setup, architecture, or usage changes.
- Record meaningful decisions in the decision log.
- Leave the project easier to understand than you found it.
- Do not introduce dependencies without a reason that survives review.
- Do not hide important behavior in unnamed conventions.

## Collaboration Rules

- Explain tradeoffs when a choice affects future maintainers.
- Call out uncertainty directly.
- Preserve context in memory files before pausing long work.
- Use review checklists before merge or release.
- Treat setup failures and unclear docs as bugs.

## Change Rules

Before a change is done, confirm:

- It solves the stated problem.
- It does not break documented workflows.
- It has appropriate tests or a clear reason tests are not needed.
- It updates docs, memory, or changelog when relevant.
- It passes the project quality bar.
