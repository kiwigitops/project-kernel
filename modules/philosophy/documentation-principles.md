# Documentation Principles

## Documentation Should Help Someone Act

Good docs answer a reader's next question.

Operational rule: every document should name its audience or use case through its content.

## Document Decisions, Not Just Outcomes

Future maintainers need to know why a path was chosen.

Operational rule: record tradeoffs when architecture, dependencies, scope, or workflow changes.

## Keep Setup Sacred

If setup docs are wrong, trust in the repo drops fast.

Operational rule: update setup instructions whenever commands, environment variables, or services change.

## Prefer Living Indexes

People should know where to start.

Operational rule: README files and `START-HERE.md` files should route readers to the next useful document.

## Remove Stale Guidance

Outdated docs are worse than missing docs because they create false confidence.

Operational rule: delete or rewrite stale sections during maintenance.
